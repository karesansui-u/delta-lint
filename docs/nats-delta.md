# Measuring δ in Nats: Design Document

**Status**: Draft / 壁打ち用
**Date**: 2026-03-27
**Paper**: "Structural Collapse as Information Loss" (Sunagawa, 2026)

## 0. このドキュメントの目的

delta-lint が検出する構造矛盾（finding）を、論文の枠組みに従って **nats（自然対数単位）** で定量化する設計を検討する。ヒューリスティックに nats のラベルを貼るのではなく、「nats で測っている」と主張できる根拠を持つ設計を目指す。

## 0.5. δ のスコープ定義 — 何を測り、何を測らないか

キャリブレーション実験の前に、δ がバグ原因全体のどこを担当するのかを明確にする。δ の重みを決めるには、まず全体像の中での位置を知る必要がある。

### 0.5.1 ソフトウェアのバグ原因の MECE 分解

```
バグの原因
│
├── A. 矛盾系（δ の領域）← delta-lint が担当
│   ├── A1. モジュール間の暗黙契約違反
│   │       例: Guard Non-Propagation, Asymmetric Defaults
│   ├── A2. 設計意図 vs 実装の乖離
│   │       例: 文書化されていない design decision が実装と矛盾
│   ├── A3. 仕様 vs 実装の乖離
│   │       例: External Spec Divergence
│   └── A4. 時間経過による劣化
│           例: Lifecycle Ordering, 古い仮定の残存
│
├── B. 複雑度系（μ/μ_c の領域）
│   ├── B1. 認知的複雑度（コードが読めない）
│   ├── B2. 状態空間爆発（並行処理、状態管理）
│   └── B3. 依存関係の深さ（変更波及の予測困難）
│
├── C. 知識不足系（N_eff の領域）
│   ├── C1. ドメイン知識の欠如
│   ├── C2. フレームワーク/ライブラリの誤用
│   └── C3. エッジケースの見落とし
│
└── D. プロセス系（方程式の外）
    ├── D1. テスト不足
    ├── D2. レビュー不足
    └── D3. デプロイ手順の問題
```

### 0.5.2 生存方程式へのマッピング

各カテゴリが `S = N_eff · (μ/μ_c) · e^{-δ}` の別の因子に対応する：

| カテゴリ | 方程式の因子 | バグを**防ぐ**手段 | バグを**見つける**手段 |
|---|---|---|---|
| A. 矛盾 | e^{-δ} | 設計の一貫性、契約の明示化 | **delta-lint** |
| B. 複雑度 | μ/μ_c | リファクタリング、モジュール分割 | 複雑度メトリクス (cyclomatic etc.) |
| C. 知識不足 | N_eff | 冗長設計、フェイルセーフ | 型システム、linter |
| D. プロセス | S_c（閾値） | CI/CD、テスト、レビュー | テストカバレッジ |

### 0.5.3 既存手法とのカバレッジ比較

| カテゴリ | 既存手法 | δ が効くか |
|---|---|---|
| A1 暗黙契約違反 | **ほぼ何もない** | **◎ δ の主戦場** |
| A2 設計↔実装乖離 | コードレビュー（属人的） | ○ |
| A3 仕様↔実装乖離 | 結合テスト（部分的） | ○ |
| A4 時間劣化 | **なし（気づかない）** | **◎** |
| B1 認知的複雑度 | linter (max-complexity) | ✗（μ/μ_c の問題） |
| B2 状態空間爆発 | model checker, sanitizer | ✗ |
| C1 ドメイン知識欠如 | 型システム, mypy | ✗（N_eff の問題） |
| C2 ライブラリ誤用 | linter rules | ✗ |
| D1 テスト不足 | カバレッジ計測 | ✗（プロセスの問題） |

**δ が最も強い領域 = A1, A4。既存手法のカバレッジがほぼゼロの領域。**

補足: カテゴリ A の矛盾は**技術的にはユニットテストで防げる**。例えば `assert file_a.DEFAULT == file_b.DEFAULT` と書けばよい。だが現実には誰も書かない — 暗黙の契約の存在に気づいていなければテストも書けない。delta-lint が矛盾を発見して初めて「ここにテストを書くべきだった」と分かる。つまり delta-lint → finding 発見 → テスト追加 が現実的なワークフロー。

### 0.5.4 δ に含めるべきもの / 含めないべきもの

```
δ に含める（矛盾系 A）:
  ✓ file_a と file_b のデフォルト値が異なる → δ > 0
  ✓ 設計書に書いてあるAPIと実装が違う → δ > 0
  ✓ イレギュラーな仕様が暗黙（文書化されていない）→ δ > 0
  ✓ Guard が片方にしか伝播していない → δ > 0

δ に含めない（他の因子の担当）:
  ✗ コードが複雑で読めない → μ/μ_c の問題
  ✗ テストがない → S_c（プロセス閾値）の問題
  ✗ 関数が長すぎる → μ の問題（認知負荷）
  ✗ CONTEXT が巨大すぎるモノレポ → μ_c 増大の問題（後述§1.6）
```

### 0.5.5 「構造死」の定義

「どこを触っても矛盾のバグが出る」状態の正体：

- S_A は高くてもよい（テストは全部通る）
- **S_B（= e^{-δ}）が崩壊している**（暗黙の仮定が壊れまくっている）
- テストが通るのにバグが出る = **テストの構造的盲点** = カテゴリ A

これが delta-lint の存在意義：**既存手法の構造的死角をカバーする**。

## 0.6. δ のフロー — 静的スナップショットから動的指標へ

δ は「ある瞬間のリポジトリの矛盾量」だけでなく、**時系列で増減する動的量**として捉える必要がある。

### 0.6.1 δ の流入と流出

```
t=0  コードベースに矛盾なし (δ=0)
t=1  新機能追加 → 暗黙の前提が1つ壊れる (δ += I₁)
t=2  レビューで気づかずリリース（δ は変わらない、未検出のまま）
t=3  バグ報告 → delta-lint が finding として検出（δ は変わらない、可視化されただけ）
t=4  fix 適用 (δ -= I₁)
```

フローの定義：

```
δ_流入 = コード変更が新たに生む矛盾の I の合計
       （新機能追加、リファクタリングの副作用、依存の更新等）

δ_流出 = fix / suppress / 文書化 が解消する I の合計
       （delta fix、明示的アノテーション、設計の一貫化等）

δ_net  = δ_流入 - δ_流出
```

### 0.6.2 δ_net によるプロジェクト健全性の分類

| δ_net の傾向 | 状態 | 解釈 |
|---|---|---|
| δ_net ≤ 0（継続的） | **健全** | 矛盾の解消速度が生成速度を上回っている |
| δ_net ≈ 0（振動） | **均衡** | 矛盾は増えも減りもしない（多くのプロジェクトのデフォルト） |
| δ_net > 0（継続的） | **構造死に向かっている** | 矛盾が蓄積。§0.5.5 の「どこを触ってもバグが出る」状態へ |

**重要**: δ_net > 0 が持続するプロジェクトは、テストが全部通っていても e^{-δ} が指数的に低下する。これは論文の「構造崩壊」の動的版。

### 0.6.3 delta-lint のフロー追跡

```
scan_history.jsonl の時系列データから:

δ_repo(t)   = Σ I(fᵢ) for open findings at time t
δ_流入(t)    = Σ I(fᵢ) for findings first detected at time t
δ_流出(t)    = Σ I(fᵢ) for findings resolved at time t
δ_net(Δt)   = δ_流入(Δt) - δ_流出(Δt)

ダッシュボードで追跡可能な指標:
- δ_repo の推移グラフ（上昇トレンド = 危険信号）
- δ_net の移動平均（正に振れ続けたらアラート）
```

### 0.6.4 「発見できずにリリースした」問題

「コード実装後に矛盾ができてそれを発見できずにリリースした」= δ_流入が delta-lint のスキャン頻度を超えている。

これは delta-lint の **watch モード** と **CI 統合** の存在意義に直結する：

```
手動 scan のみ:   δ_流入の検出に遅延 → 矛盾がリリースに含まれるリスク
watch モード:     ファイル保存時に即座にスキャン → δ_流入をリアルタイム検出
CI 統合:          PR マージ前にスキャン → δ_流入がリリースに到達する前に捕捉
```

つまり CI gate の fail_severity は「δ_流入のうち I > threshold のものをリリースに入れない」というフロー制御。

### 0.7 重みの根拠

δ の重みはカテゴリ A の**中で**しか意味を持たない。パターン別の I_base の比率は「矛盾系バグ全体の中でそれぞれが占める割合」から決まるべき。

実測データの候補：
- delta fix で修正した finding → その後バグが再発したか
- suppress した finding → suppress 後にその領域でバグが出たか
- finding のパターン別に、fix 後のバグ発生率の変化

→ キャリブレーション実験（§5）で LLM accuracy を使うのは、上記「矛盾がバグをどれだけ生むか」のプロキシとして「矛盾が推論をどれだけ狂わせるか」を計測するもの。

## 1. 論文が言っていること

### 1.1 核心の方程式

```
S = N_eff · (μ/μ_c) · e^{-δ}          ... Eq. 2

δ = Σᵢ I(Cᵢ)                          ... Eq. 5

I(Cᵢ) = -ln(1 - qᵢ) = -ln(pᵢ)  [nats]  ... Eq. 4
```

- `qᵢ` = 制約 Cᵢ が除去する構成空間の割合
- `pᵢ` = 制約 Cᵢ を満たす構成の割合
- `δ` = 累積情報損失（全制約の自己情報量の和）
- `e^{-δ}` = SAT の第一モーメントと**数学的に同一**

### 1.2 二つのレベル

| レベル | δ の計算方法 | 精度 | 例 |
|---|---|---|---|
| **Level A** | 系の外部で定義された survival、δ は直接計算可能 | 定量的に鋭い | SAT（q = 問題構造から計算）、LLM 実験（accuracy で直接計測） |
| **Level B** | 系自身の同一性に依存する survival、δ はプロキシ計測 | 計測誤差 ε に応じて劣化 | 企業、生態系、**コードベース** |

### 1.3 SAT での3アンカーポイント

| 制約タイプ | I (nats) | q (除去割合) | 根拠 |
|---|---|---|---|
| Random 3-clause | ln(8/7) ≈ 0.134 | 1/8 | 3変数のうち1割当のみ不満足 |
| XOR pair | ln 2 ≈ 0.693 | 1/2 | 2変数の半分の割当を除去 |
| Implication chain (L=4) | 4ln2 - ln4 ≈ 1.39 | 1 - 4/2⁴ = 3/4 | L パターン + (L-1) 境界割当 |

これらは**第一原理から計算**された値。フィッティングパラメータなし。

### 1.4 LLM 実験での計測

論文 §5.3 (Exp. 5) で δ の次元構造を実験的に確認:

```
δ_factual (L0-L3):  accuracy 維持 → I ≈ 0 に近い
δ_structural (L4-L6): accuracy → 0% → I が大きい（系を壊す）
```

LLM では I は input からではなく **output から逆推定**:

```
I_i = -ln(S_i / S_0)
```

### 1.5 コードベースで何が問題か

論文 §6.7:
> "the primary open challenge is developing domain-specific methods to calibrate δ in nats — that is, to estimate the fraction of viable configurations eliminated by each constraint, which requires identifying both the relevant configuration space and the appropriate margin variable (μ) for each domain."

**2つの未解決問題:**

1. **構成空間が定義されていない**: SAT は 2ⁿ 個の割当。コードベースの「viable な構成」とは？
2. **除去割合 q が計算できない**: finding ④ が構成空間の何割を除去するか不明

### 1.6 δ の再定義 — 「期待からの乖離」としての KL ダイバージェンス

§1.5 の未解決問題に対する回答。SAT の構成空間 `{0,1}^n` に対応するものは「コードの状態空間」ではなく：

> **「読み手（LLM/人間）がコードベースについて持つ事前期待」と「実際のコード」の乖離**

つまり：

```
δ = D_KL( P_actual || P_expected )
```

これは論文 §6.2 Result 1 そのもの：

```
δ = D_KL( P_SAT || P_0 )
  = 「制約付き分布と制約なし分布の KL ダイバージェンス」
```

「制約が除去する構成空間の割合」ではなく「**期待からの乖離**」として δ を定義すると、コードベースでの具体例がすべて整合する：

| 状況 | δ が大きい理由 |
|---|---|
| 設計書と実装の矛盾 | P_expected（設計書を読んだ期待）と P_actual（実装）が乖離 |
| イレギュラーな仕様が文書化されていない | 読み手の期待を補正する情報がない → 乖離が大きい |
| イレギュラーな仕様が**明示されている** | CLAUDE.md や docstring で期待が補正済み → **δ ≈ 0** |
| Dead code | 「使われている」という期待 vs 実際は未使用 → 小さい乖離 |
| Guard Non-Propagation | 「file_a にガードがあるなら file_b にもある」という期待 vs 実際はない → 大きい乖離 |

#### 「文書化すれば δ ≈ 0」の意義

これは SAT では起きない現象（制約は制約のまま）。だがコードベースでは**期待を事前に補正する**ことで δ を減らせる。

- `// ⚠ ここは設計と異なる: 仕様上意図的に非対称` → 読み手の期待が補正される → δ ≈ 0
- delta-lint の suppress コメント = まさにこれ
- **delta-lint が finding を報告した時点で、読み手にとってその finding の δ は減少する**

つまり delta-lint は δ を「検出」するだけでなく、報告すること自体が δ を**低減**する道具でもある。

#### 暗黙知の蓄積 = δ を能動的に下げる運用

「文書化すれば δ ≈ 0」は単発の操作ではなく、**継続的な運用**として理解すべき。

コードだけ見てもわからない情報（暗黙の前提、過去バグの経緯、設計判断の理由）は、文書化されていない限り δ > 0 のまま残る。これを δ 低減の観点で整理すると：

| 行為 | δ への効果 | 例 |
|---|---|---|
| CLAUDE.md にイレギュラーな仕様を記載 | δ ≈ 0（LLM の期待を事前補正） | 「`timeout` は file_a と file_b で意図的に異なる」 |
| docstring / コメントに設計判断の理由を記載 | δ ≈ 0（読み手の期待を補正） | 「歴史的理由で非対称 API」 |
| delta-lint の finding 報告 | δ_読み手 ≈ 0（乖離の可視化） | finding レポートを読んだ開発者は矛盾を認識済み |
| delta fix でコードを修正 | δ_コード = 0（矛盾自体を解消） | Guard を伝播させて一貫化 |
| suppress + 理由コメント | δ ≈ 0（「これは許容された乖離」と明示） | `# delta-lint:suppress — 仕様上意図的` |

**暗黙知を蓄積していく営み全体が δ の継続的低減**。個別の finding を fix するだけでなく、「コードベースの暗黙の前提をどれだけ明文化しているか」がプロジェクトの δ_repo を決める。

これは §0.6 のフロー概念と合わせると、δ_流出には fix だけでなくドキュメンテーションも含まれることを意味する：

```
δ_流出 = Σ I(fix されたfinding) + Σ I(文書化で期待補正されたfinding)
```

#### 乖離の多層モデル

δ は単一の層ではなく、複数の「期待の層」に存在する：

```
Layer 0: 設計意図 (design intent)     ─┐
Layer 1: 仕様書 (spec)                 ├─ 各層間に δ が存在
Layer 2: コード (implementation)       │
Layer 3: LLMの事前期待 (LLM's prior) ─┘
```

- **設計 ↔ コード**: 本来こうあるべきが実装に反映されていない
- **仕様 ↔ コード**: spec に書いてあることとコードが違う
- **コード ↔ LLMの期待**: LLM が「普通こうだろう」と推論する状態と実際が違う
- **仕様 ↔ LLMの期待**: イレギュラーな仕様が文書化されていない → LLM が誤推論

**計測可能性**: Layer 3（LLM期待 ↔ 実態）は Level A で直接測れる。LLM に推論させて正答率低下を見ればよい。他の層は Level B 寄り。

#### モノレポ / 巨大 CONTEXT の問題

「CONTEXT が大きすぎるのもあるべき状態ではない」は正しいが、これは δ ではなく **μ/μ_c** の問題：

```
μ   = 読み手が使える処理能力（コンテキストウィンドウ、注意力）
μ_c = コードベースを正しく理解するのに必要な最小処理能力

モノレポで CONTEXT 巨大 → μ_c 増大 → μ/μ_c 低下 → S 低下
```

論文 Exp. 3-5 がこれを示している：δ=0 でも μ（コンテキスト長）を圧迫すると accuracy は 1.00 → 0.70 に緩やかに低下。**ただし崩壊は起きない**。崩壊を起こすのは δ > 0 のとき。

```
CONTEXT 巨大のみ = μ_c 増大 → 緩やかな劣化
矛盾ありのみ     = δ 増大   → 急激な崩壊の可能性
両方              = S = μ/μ_c · e^{-δ} の急落（乗法的）
```

## 2. 「nats で測る」とはどういう意味か

### 2.1 nats でないもの

```python
# これは nats ではない — ヒューリスティックにラベルを貼っただけ
PATTERN_I_BASE = {"④": 0.69, "⑦": 0.13, ...}
```

パターンの順序（④ > ⑦）は正当化できるが、絶対値（0.69 nats）は根拠がない。「XOR っぽいから ln 2 に近いだろう」はアナロジーであって計測ではない。

### 2.2 nats であるための条件

`I(C) = -ln(p)` が nats であるためには:

1. **構成空間** Ω が定義されている
2. Ω 上の**確率測度** P が定義されている
3. 制約 C を満たす部分集合の測度 `p = P({ω ∈ Ω : ω satisfies C})` が**計測可能**

SAT はこれを完全に満たす（Ω = {0,1}ⁿ, P = 一様分布, p = |SAT|/2ⁿ）。

### 2.3 コードベースの構成空間の候補

| 構成空間の定義 | p の計測方法 | 問題 |
|---|---|---|
| テストスイートの pass/fail パターン | 制約追加前後の pass率変化 | テストが矛盾を捕捉できない場合 p=1 のまま |
| LLM の推論正答率 | 矛盾あり/なしで accuracy 比較 | Level A に準ずるが、コスト大 |
| コード変更の受容可能集合 | 制約に違反しない diff の割合 | 定義が曖昧、計測困難 |
| リポジトリのフォーク/ブランチ群 | 矛盾を含まないフォークの割合 | 母集団が小さすぎる |

## 3. 計測可能な道

### 3.1 道A: LLM-as-Instrument — fix ベース（旧案）

論文 Exp. 5 のプロトコルをそのまま適用する。

**原理**: LLM の推論精度は外部定義された survival（正答率）。コンテキストに finding の矛盾を含めて推論タスクを出し、accuracy の変化から I を逆推定する。

```
プロトコル:
1. finding f のコンテキスト（file_a, file_b の関連コード）を取得
2. そのコンテキストに基づく推論タスクを生成
   - 例: 「このコードで X を呼び出すと何が返るか？」
3. δ=0 条件: 矛盾を解消したコードで同じタスク
4. δ>0 条件: 矛盾を含むコードで同じタスク
5. 各条件 n=30 trials、accuracy を計測
6. I(f) = -ln(acc_δ>0 / acc_δ=0)
```

**問題**: fix が正しい保証がない。fix を適用した状態が本当に δ=0 かどうか検証できない。

### 3.2 道A': LLM-as-Instrument — アノテーションベース（推奨案） ⭐

§1.6 の「期待からの乖離」定義に基づく改良版。fix を適用するのではなく、**乖離の存在を明示する**ことで期待を補正し、δ=0 条件を作る。

```
プロトコル:
1. finding f のコンテキスト（file_a, file_b の関連コード）を取得
2. そのコンテキストに基づく推論タスクを生成
   - 例: 「このコードで X を呼び出すと何が返るか？」

3. 条件A（δ>0）: コードをそのまま見せて推論タスク
   → 矛盾が暗黙に存在する → LLM の期待と実態が乖離 → accuracy 低下

4. 条件B（δ≈0）: 矛盾箇所に明示的アノテーションを付与して同じタスク
   → "⚠ 注意: file_a のデフォルトは X だが、file_b のデフォルトは Y（仕様上意図的）"
   → 読み手の期待が補正される → accuracy 回復

5. 各条件 n=30 trials、accuracy を計測
6. I(f) = -ln(acc_A / acc_B)
```

**なぜこちらが優れるか:**

1. **δ=0 条件の作りやすさ**: fix を適用するのではなく、「乖離があることを明示する」だけ。fix の正しさに依存しない
2. **§1.6 との整合性**: δ = 「暗黙の乖離」。明示すれば δ ≈ 0。これは定義そのもの
3. **delta-lint の実際の動作と一致**: finding を報告する = 「ここに期待との乖離がありますよ」と明示する。つまり delta-lint が finding を報告した時点で、その finding の δ は（読み手にとって）減少する
4. **CLAUDE.md 運用との一致**: イレギュラーな仕様を CLAUDE.md に書く = 期待を事前補正 = δ 低減

**コスト見積もり**: 道A と同じ（パターンタイプ別で ~900 calls、claude -p で $0）

### 3.3 道B: Fix-Effect Measurement（修正効果プロキシ）

`delta fix` で finding を修正し、下流への影響を計測。

```
プロトコル:
1. finding f の fix を生成（fixgen.py）
2. fix 適用前: 依存モジュールの静的解析スコア S_before
   - import 先のコードとの整合性チェック（型、API 契約等）
3. fix 適用後: 同じスコア S_after
4. I(f) = -ln(S_before / S_after)  — ただし S は [0,1] に正規化
```

**問題**: 「静的解析スコア」が well-defined でない。循環の危険。

### 3.4 道C: パターンタイプ別 LLM キャリブ + 文脈補正

**Phase 1: I_base テーブルを LLM 実験で一度だけ実測**

道A'（アノテーションベース）のプロトコルを使用：

```
10パターン × 3重大度 × 30 trials = 900 LLM calls（claude -p で $0）

各パターンの代表的な finding を用意:
- ④ Guard Non-Propagation: バリデーションが片方にしかない
- ① Asymmetric Defaults: 初期値が file_a と file_b で異なる
- ⑦ Dead Code: 未使用関数が存在
- ...

条件A: 矛盾を含むコードをそのまま見せて推論
条件B: 矛盾箇所にアノテーション付与して推論
  → I(pattern, severity) = -ln(acc_A / acc_B) [nats]
```

**Phase 2: 個別 finding は I_base のみ（φ=1.0 推奨）**

```
I(f) = I_base(pattern, severity)

fan_out, churn 等は δ ではなく S の別因子（N_eff, μ/μ_c）で扱う。
§1.6 の「期待からの乖離」定義では、同じパターン×同じ重大度なら
読み手の期待を裏切る度合いは同じ。影響の広さは別の問題。
```

**ここでの正当化:**
- `I_base` は LLM 実験で**実測**した nats 値 → 根拠あり
- φ=1.0 は論文的に正しい（fan_out は N_eff の問題、§7.1 参照）
- アノテーションベースのプロトコルは「明示すれば δ ≈ 0」という定義に直接対応

### 3.5 道D: 順序のみ使い、絶対値は「未キャリブレーション」と明示

```
δ_repo = Σ w(fᵢ)  [uncalibrated units]

ただし順序は保証:
w(④, high) > w(①, high) > w(②, medium) > ... > w(⑦, low)
```

正直だが、「nats で測る」という目的は達成できない。

## 4. 構成空間の問題を深掘りする

### 4.1 SAT との対応を真剣に考える

SAT:
```
Ω = {0,1}ⁿ                    — n 変数の全割当
C = clause                     — 割当の部分集合を除去
q = |violating assignments| / 2ⁿ  — 除去割合
I(C) = -ln(1-q)               — 自己情報量
```

コードベースで対応するものは？

```
Ω = ?                          — 「コードベースの状態空間」
C = finding                    — 構造矛盾
q = ?                          — 矛盾が除去する「viable 状態」の割合
I(C) = -ln(1-q)               — 計測したい量
```

### 4.2 構成空間の候補: 「暗黙契約の充足可能性」

delta-lint が検出するのは「暗黙の契約」の違反。契約を boolean 制約とみなせば:

```
Ω = {コードベース状態の集合}
契約 = 「file_a の default と file_b の default が一致する」等
finding = 「契約 C が違反されている」
```

ここで q は「ランダムなコード変更が契約 C を満たす確率」の補数:

```
q = 1 - P(random edit satisfies C)
```

**問題**: 「ランダムなコード変更」の分布が定義できない。

### 4.3 構成空間の候補: 「開発者の判断空間」

finding が存在するとき、開発者が下す判断の空間:

```
Ω = {この矛盾に遭遇した開発者が取りうる行動}
  = {file_a に合わせる, file_b に合わせる, 両方変える, 無視する, ...}

正解は1つ。finding の存在が正解の判別を困難にする度合いが I(f)。
```

これは論文の LLM 実験と**直接対応する**: LLM を開発者のプロキシとして使い、矛盾ありの条件で正答率が下がる → 情報損失。

### 4.4 たどり着く結論

**最も理論整合的な構成空間は「推論空間」**:
- Ω = LLM（or 開発者）が生成しうる回答の集合
- p = 矛盾が存在する条件で正解を出す確率
- I(f) = -ln(p_with_contradiction / p_without)

これは**論文の LLM 実験そのもの**であり、Level A に近い計測が可能。

§1.6 の「期待からの乖離」定義と統合すると：

```
δ = D_KL( P_actual || P_expected )

Ω  = LLM/開発者の推論空間
P_expected = 矛盾が明示されている（アノテーション済み）条件での推論分布
P_actual   = 矛盾が暗黙に存在する条件での推論分布

I(f) = -ln(acc_暗黙 / acc_明示)
     = 「暗黙の矛盾が推論にもたらす情報損失」[nats]
```

これにより §3.2（道A': アノテーションベース）のプロトコルが理論的に正当化される。

## 5. 推奨アーキテクチャ

### 5.1 単層構造（φ=1.0 推奨）

```
┌─────────────────────────────────────────────────────┐
│  I_base テーブル（LLM アノテーション実験で実測）         │
│                                                       │
│  パターン × 重大度 → I [nats]                          │
│  計測方法: 道A'（アノテーションベース）プロトコル         │
│  頻度: delta-lint のパターン定義変更時に再キャリブ        │
│  精度: Level A（LLM accuracy は外部定義 survival）      │
│                                                       │
│  I(f) = I_base(pattern, severity)                     │
│  φ=1.0: fan_out 等は δ に含めない（N_eff で扱う）       │
└─────────────────────────────────────────────────────┘
```

φ≠1.0 の文脈補正は Phase 2 以降で N_eff・μ/μ_c を実装する際に、適切な因子に割り当てる。δ の純度を保つために Layer 1 のみで運用する。

### 5.2 キャリブレーション実験の設計（アノテーションベース）

§1.6 の「期待からの乖離」定義に基づく設計。fix の正しさに依存しない。

```
実験マトリクス: 10 パターン × 3 重大度 = 30 セル
各セル: n = 30 trials

手順（各セルごと）:
1. パターン P、重大度 S の代表的 finding を選定
2. finding のコンテキスト（file_a, file_b + 周辺コード）を抽出
3. コンテキストに基づく推論タスクを設計
   - 「この関数を X で呼ぶと何が返るか」
   - 「このAPIの正しい使い方はどれか」等

4. 条件A（δ>0）: コードをそのまま見せて推論 → acc_A
   - 矛盾は暗黙に存在（コメントやアノテーションなし）

5. 条件B（δ≈0）: 矛盾箇所にアノテーションを付与して推論 → acc_B
   - 例: "⚠ 注意: file_a のデフォルトは timeout=30 だが、
           file_b のデフォルトは timeout=60（仕様上意図的な非対称）"
   - 読み手の期待を事前補正する

6. I(P, S) = -ln(acc_A / acc_B) [nats]

コスト: 30 セル × 60 calls = 1,800 calls（claude -p で $0）
所要時間: 並列実行で ~2時間
```

**なぜ「アノテーション」が δ=0 条件として有効か:**

- δ = 「暗黙の乖離」が読み手の推論に与える情報損失（§1.6）
- アノテーション = 乖離の存在を明示 = 期待を補正 = δ ≈ 0
- これは CLAUDE.md に注意書きを足す操作と同じ
- delta-lint が finding を報告すること自体が、この操作に相当する

### 5.2.1 Phase -1 パイロット実験の結果（2026-03-27 実施）

Phase 0 の本実験に先立ち、④ Guard Non-Propagation × high の 1 セルでパイロットを実施。

#### 実験設計の反復

**V1/V2（全コンテキスト方式）: I = 0 nats**

§5.2 のプロトコルをそのまま適用 — 矛盾を含む全ファイルを条件A/B 両方で提示。

```
V1: 3ファイル、バリデーション有無 → Sonnet 30/30 正解、I = 0
V2: 8ファイル、timeout 非対称   → Sonnet 30/30、Haiku 30/30、I = 0
```

**発見**: LLM は提供されたコード全体を等しく注意深く読む。矛盾が「そこに書いてある」限り、アノテーションの有無は accuracy に影響しない。

これは §5.2 のプロトコルの本質的な限界を示す: **全ファイルを提示すれば δ_measured ≈ 0 になる。**

**V3b（部分コンテキスト方式）: I = 1.099 nats ⭐**

矛盾の片方のファイルを**提示しない**方式に変更。人間の開発体験をシミュレート。

```
シナリオ: ユーザー管理 API
  visible: DELETE /users/:id → user_service.remove_user() → 204
  hidden:  user_service.remove_user() = ソフトデリート（is_active=False + email suffix 変更）

プロトコル:
  条件A（δ>0）: hidden file なし。LLM は DELETE=物理削除と仮定 → 誤推論
  条件B（δ≈0）: visible file にアノテーション追加 → 正推論

設計: 3質問 × 10試行 × 2条件 = 60 calls, temperature=0.7
```

| モデル | acc_A | acc_B | **I(f) [nats]** | e^{-I} |
|---|---|---|---|---|
| claude-sonnet-4-20250514 | 0.333 | 1.000 | **1.099** | 0.333 |
| claude-haiku-4-5-20251001 | 0.333 | 1.000 | **1.099** | 0.333 |

質問別:

| 質問 | 条件A | 条件B | 内容 |
|---|---|---|---|
| Q1 (DBクエリ) | 0/10 (全員 A:物理削除) | 10/10 | DELETE後にレコードが残るか |
| Q2 (再登録) | 10/10 | 10/10 | 同じメールで再登録できるか |
| Q3 (GDPR) | 0/10 (全員 C:監査ログ) | 10/10 | 個人データが完全に消えたか |

#### 知見

1. **プロトコルの修正が必要**: §5.2 の「全ファイル提示 + アノテーション」では δ ≈ 0 になる。Phase 0 では**部分コンテキスト方式**を採用すべき。矛盾の片方のファイルを hidden にし、アノテーションで「hidden file の挙動」を伝える。

2. **反直感的実装でないと差が出ない**: 一般知識から推論可能な矛盾（HTTP timeout の挙動等）では条件A でも正解できる。δ > 0 が測定できるのは、**コード固有の実装詳細を知らないと正解できない**質問に限る。

3. **モデルサイズに非依存**: Sonnet と Haiku で完全同一の結果。これは「コード読解能力」ではなく「情報の有無」が accuracy を支配していることを示す。δ は読み手の能力ではなく、情報の暗黙性そのものを計測している。

4. **I = 1.099 nats の解釈**: ln(3) ≈ 1.099。3つの選択肢から1つしか正解できない状態。XOR 制約（ln 2 = 0.693）を超える重大な情報損失。ソフトデリートの暗黙性は、構造的整合性を e^{-1.099} = 33.3% まで低下させる。

5. **Phase 0 への示唆**: 全 30 セル（10パターン × 3重大度）で部分コンテキスト方式を使用。各セルの質問は「hidden file の具体的実装を知らないと正解できない」ものを設計する。予想される結果:
   - high severity: I ≈ 0.5-1.5 nats
   - medium severity: I ≈ 0.1-0.5 nats
   - low severity: I ≈ 0.0-0.1 nats

#### プロトコル改訂

```
改訂版プロトコル（Phase 0 で使用）:

1. finding f のコンテキストを取得（file_a, file_b + 周辺コード）
2. file_b を hidden にする
3. file_a + 他の visible files のみでの推論タスクを設計
   - file_b の具体的実装を知らないと正解できない質問
   - 一般知識では正解を確定できない質問

4. 条件A（δ>0）: visible files のみ提示 → 推論 → acc_A
5. 条件B（δ≈0）: visible files + file_a にアノテーション付与 → 推論 → acc_B
   - アノテーション = hidden file の矛盾に関する情報を明示

6. I(P, S) = -ln(acc_A / acc_B) [nats]
```

スクリプト: `plugins/delta-lint/scripts/experiments/pilot_v3b.py`（再利用可能な実験基盤）

### 5.3 バグ原因帰属フレームワーク — キャリブレーションの実データ源

§0.5 の MECE 分解を実運用に落とすと、**バグが発生するたびに原因を生存方程式の因子に帰属させる**フレームワークが得られる。これは Phase 0 のキャリブレーション実験とは別系統の、より「リアル」なデータ源になりうる。

#### バグ発生時の分類プロトコル

```
バグが発生したら以下を判定:

(A) 情報不足 → δ の問題
    「file_a のデフォルトが 30 だと書いてなかった」
    「この API は歴史的理由で非対称」が伝わっていなかった
    → 説明を足せば次から防げたか？ → Yes なら δ
    → 暗黙の前提を明示する = δ を下げる

(B) LLM/開発者の能力限界 → μ/μ_c の問題
    「説明は全部あったが、コードが複雑すぎて間違えた」
    「コンテキストが長すぎて注意が散漫になった」
    → 説明を足しても防げない → コード簡素化 or モジュール分割

(C) 知識不足 → N_eff の問題
    「そもそもこのフレームワークの使い方を知らなかった」
    「ドメイン固有の制約を知らなかった」
    → 型/linter/ドキュメントで防ぐ領域
```

#### なぜこれがキャリブレーションに有用か

LLM アノテーション実験（§5.2）は **人工条件** での計測。バグ原因帰属は **実環境** でのデータ。両者を突き合わせることで：

1. **I_base の妥当性検証**: パターン P の I_base が大きいなら、カテゴリ A に帰属されるバグも P 関連が多いはず
2. **カテゴリ境界の曖昧さの解消**: 実際のバグで「A か B か迷う」ケースが蓄積 → 分類基準の精緻化
3. **LLM プロキシの限界検出**: LLM では I ≈ 0 だが人間には δ が大きいケース → LLM が見抜けない矛盾パターンの特定

#### データの蓄積方法

```
findings.jsonl の既存フィールドを拡張:

{
  "id": "dl-a3f2c1",
  "pattern": "④",
  "severity": "high",
  ...
  "bug_attribution": {           // バグ発生時に追加
    "bug_id": "GH-1234",        // GitHub Issue 等
    "category": "A",             // A=δ, B=μ/μ_c, C=N_eff
    "rationale": "validation の非対称性が文書化されていなかった",
    "preventable_by_doc": true   // 説明を足せば防げたか
  }
}
```

これは append-only ログの設計（CLAUDE.md §アーキテクチャ制約）と整合する。

#### LLM コーディング時の帰属

LLM にコードを書かせてバグが出た場合の特別な分類：

| 状況 | 帰属 | δ との関係 |
|---|---|---|
| 暗黙の前提が LLM に伝わっていなかった | A（δ） | CLAUDE.md / コメントに書けば防げた |
| コンテキストに全情報あったが LLM が見落とした | B（μ/μ_c） | コンテキスト長 or 複雑度の問題 |
| LLM が API/フレームワークの使い方を知らなかった | C（N_eff） | 知識の問題（モデル更新で改善しうる） |
| LLM の変更が新たな矛盾を生んだ（他ファイルとの整合性破壊） | A（δ、新規流入） | §0.6 の δ_流入。scan/CI で捕捉すべき |

最後のケース（実装後に矛盾が生まれる）は特に重要。LLM が生成したコードが既存コードと矛盾する = δ_流入。これは delta-lint の CI gate で捕捉すべき流入であり、「発見できずにリリースした」場合は §0.6.4 の問題に帰着する。

### 5.4 集約メトリクス

```python
@dataclass
class DeltaMetrics:
    """リポジトリの δ メトリクス"""

    # Per-finding
    finding_deltas: dict[str, float]   # finding_id → I(f) [nats]

    # Aggregate
    delta_total: float                 # Σ I(fᵢ) [nats] — open findings のみ
    delta_structural: float            # Σ I(fᵢ) for ①②④⑨
    delta_computational: float         # Σ I(fᵢ) for ③⑤⑥
    delta_factual: float               # Σ I(fᵢ) for ⑦⑧⑩

    # Survival potential（μ/μ_c = 1.0 固定、N_eff = 1.0 固定）
    # → Phase 1 では e^{-δ} のみ
    survival_factor: float             # e^{-δ_total}

    # Calibration metadata
    calibration_source: str            # "llm-experiment-v1" or "uncalibrated"
    i_base_table: dict[str, float]     # (pattern, severity) → I [nats]
```

```
  3. dl-c9d3f7 ② med   0.41 nats  (I_base=0.39, φ=1.05)
```

### 5.6 δ_repo ヘルスバロメーター — 「やばい / 大丈夫」の判断基準

δ_repo の nats 値だけでソフトウェアの構造的健全性を直感的に判断できるスケールを定義する。

#### e^{-δ} による健全性スケール

e^{-δ} は「構造的整合性の残存率」。1.0 = 矛盾なし、0 に近づくほど崩壊。

| δ_repo (nats) | e^{-δ} | 判定 | 意味 |
|---|---|---|---|
| 0 – 0.5 | 1.00 – 0.61 | 🟢 **健全** | 矛盾はほぼない。通常の開発で問題なし |
| 0.5 – 1.5 | 0.61 – 0.22 | 🟡 **注意** | 矛盾が蓄積し始めている。個別 fix で対処可能 |
| 1.5 – 3.0 | 0.22 – 0.05 | 🟠 **警告** | 構造的整合性が急速に低下。重点的な fix or リファクタリングを検討 |
| 3.0 – 5.0 | 0.05 – 0.007 | 🔴 **危険** | 構造的整合性がほぼ崩壊。大規模な構造改善が必要（§7.4 diagnose） |
| 5.0+ | < 0.007 | 💀 **構造死** | どこを触っても矛盾に当たる状態。§0.5.5 の「構造死」 |

#### 閾値の根拠

これらの閾値は Phase 0 キャリブレーション後に調整が必要だが、初期値の根拠：

- **0.5 nats** ≈ XOR 制約1個分（ln 2 ≈ 0.693）。矛盾が1件あっても軽微なら影響は小さい
- **1.5 nats** ≈ e^{-δ} = 0.22。構造的整合性が 1/5 以下 → 5回に4回は矛盾に遭遇する確率
- **3.0 nats** ≈ e^{-δ} = 0.05。整合性 5%。論文の SAT 実験で崩壊が起きる領域
- **5.0 nats** ≈ e^{-δ} = 0.007。実質ゼロ。修正より作り直しの方が早い可能性

#### δ_net との組み合わせ

δ_repo の絶対値だけでなく、δ_net（§0.6）のトレンドを組み合わせることでより正確な診断が可能：

```
                δ_net ≤ 0          δ_net > 0
              （改善傾向）        （悪化傾向）
δ_repo 低     🟢 健全             ⚠ 注意（悪化し始めている）
δ_repo 高     📈 回復中           🔴 構造死に向かっている
```

#### 出力例（バロメーター付き）

```
═══ δ-lint: Repository Health ═══

  δ_repo = 2.31 nats   🟠 WARNING
  e^{-δ} = 0.099       structural integrity: 9.9%

  δ_net (30d) = +0.42 nats/month  📈 worsening
  → At this rate, δ_repo will reach 🔴 CRITICAL in ~1.6 months

  Action: 3 findings share root cause (Module A interface) — consider refactoring
          Run `delta-lint diagnose` for detailed analysis
```

この出力だけで「このリポジトリは今やばいのか大丈夫なのか」「放置するとどうなるか」「何をすべきか」が分かる。

#### 規模による正規化（検討中）

δ_repo の絶対値はリポジトリの規模に依存する。1000 ファイルのリポで δ=3.0 と 10 ファイルのリポで δ=3.0 は深刻さが違う。

正規化の候補：
- δ_repo / モジュール数 = モジュールあたりの矛盾密度
- δ_repo / LoC（千行あたり）= コード密度あたりの矛盾量

→ Phase 2 で N_eff を実装する際に正規化方法も決定する。Phase 1 では絶対値のみ。

## 7. 未解決の問い

### 7.1 φ(context) は必要か？

`I(f) = I_base × φ` の `φ` を入れると、I_base の実測値にヒューリスティックを掛けることになる。`φ=1.0`（全 finding を I_base のみで評価）の方が理論的に正直。

- **φ=1.0 派**: パターン P、重大度 S が同じなら情報損失は同じ。fan_out の違いは N_eff 側（survival equation の別の因子）で吸収すべき。
- **φ≠1.0 派**: 同じ Pattern ④ でも、fan_out=50 のファイルの方が「より多くの構成を制約する」のは直感的に正しい。

→ **論文的には φ=1.0 が正しい。fan_out は N_eff の問題。** δ は純粋に「制約の情報内容」であり、影響範囲は survival equation の他の因子が担当する。

### 7.2 N_eff と μ/μ_c をいつ測るか？

Phase 1 では δ のみ計測し、S = e^{-δ} を報告する（N_eff = μ/μ_c = 1.0 固定）。

**N_eff の候補**:
- モジュール数（独立コンポーネント数）
- 分岐の数（OR ゲート構造 = 冗長パス）
- テストカバレッジ（検証済み状態の数）

**μ/μ_c の候補**:
- CI パイプライン深度
- レビューカバレッジ
- 論文の LLM 実験では context length ratio

→ **これは Phase 2 以降。まず δ を正しく測ることに集中する。**

### 7.3 キャリブレーション実験の妥当性

LLM の推論精度変化をコードベースの δ のプロキシとして使う根拠は何か？

- **正当化**: 論文が LLM 実験で δ を測る根拠と同じ。LLM は「制約処理系」であり、構造矛盾が推論に与える影響は I(C) に比例する。
- **限界**: LLM の推論は人間の開発判断の近似。LLM が捕捉できない矛盾（超ドメイン固有等）は I が過小評価される。
- **テスト可能な予測**: キャリブレーション実験で得た I_base の順序が、実際の fix 後のバグ発生率変化の順序と一致するか検証可能。

### 7.4 マクロ診断 — findings 群からアーキテクチャ問題を検出できるか？

delta-lint の scan はファイルペア単位の「顕微鏡」。だが findings が蓄積すると、個別の矛盾ではなく**アーキテクチャレベルの設計問題**が浮かび上がることがある：

```
症状（scan が検出）:
  パターン④ が A↔B, A↔C, A↔D, A↔E の境界で計 18 件

診断（diagnose が推論すべきこと）:
  「モジュール A のインターフェース設計が破綻している」
  「個別に 18 件 fix するより、A のインターフェースを再設計した方が
   δ_流出が大きく、以後の δ_流入も止まる」
```

これは δ の計測自体ではなく、**計測結果のメタ分析**。必要な推論：

| 分析 | 入力 | 出力 |
|---|---|---|
| クラスタリング | 全 findings のパターン・モジュール分布 | 「同じ根本原因を共有する findings 群」 |
| 根本原因推定 | findings 群 + モジュール構造 + 依存グラフ | 「根本原因はモジュール A の設計」 |
| 修正戦略比較 | 個別 fix の δ_流出 vs 構造改善の δ_流出 | 「リファクタリングの方が ROI 高い」 |
| δ_流入予測 | 過去の δ_流入パターン + 構造変更案 | 「構造改善で今後の δ_流入が止まる」 |

**「大きな修正で、あとから楽になるならやったほうがいい」**— これを定量的に判断するには：

```
現状維持コスト:
  δ_流入(予測) × T期間 = 今後蓄積する矛盾の総量
  + 個別 fix コスト × finding 数

構造改善コスト:
  リファクタリングの工数（一時的に大きい）
  - δ_流入(改善後) × T期間 の削減分

→ 損益分岐点が T < 妥当な期間 なら「大改造すべき」
```

実装イメージ: `delta-lint diagnose` コマンド。全 findings + モジュール構造を LLM に渡し、上記のメタ推論を実行。scan（顕微鏡）に対する「X 線写真」。

→ **Phase 4 以降。データ（findings.jsonl）は既にある。足りないのは集約・解釈レイヤー。**

### 7.5 Chao1 との統合

現在の `info_theory.py` の Chao1 推定（未発見 finding 数の推定）は δ と直交する:

```
Chao1 → 「まだ見つかっていない finding がいくつあるか」
δ → 「見つかった finding の合計情報損失」

統合:
δ_estimated_total = δ_observed + δ_unseen
δ_unseen ≈ (unseen_count) × mean(I_base)   — Chao1 推定数 × 平均情報損失
```

これにより「リポの真の δ はどれくらいか」の推定が可能。論文のカバレッジ概念との対応。

## 9. FAQ

**Q: 結局 nats で測れるの？**

A: YES。§1.6 で δ を「期待からの乖離」= D_KL(P_actual || P_expected) と再定義したことで、構成空間の問題が解消された:
- 構成空間 Ω = LLM/開発者の推論空間（明確に定義可能）
- p = アノテーションなし（暗黙の矛盾）条件での正答率
- p₀ = アノテーションあり（期待補正済み）条件での正答率
- I(f) = -ln(p / p₀) [nats] — 測定可能
- 論文自身が LLM 実験を Level A として扱っているので、同じプロトコルで得た値は同じ精度を主張できる

**Q: φ(fan_out) を入れたら nats じゃなくなる？**

A: 厳密にはそう。I_base が nats で、φ が無次元係数なら、積の単位は nats のまま。だが「I_base × φ が情報損失を表す」根拠が φ=1.0 のときより弱くなる。φ は Level B のプロキシ補正であることを明示すべき。→ 推奨: φ=1.0 で、fan_out は S の別因子（N_eff）で扱う。

**Q: scoring.py の ROI スコアは不要になる？**

A: 不要にならない。δ と ROI は質問が違う:
- δ = 「この矛盾がシステムに与える損害」（情報理論）
- ROI = 「この矛盾を直す価値」（コスト/ベネフィット）
- 高 δ でも fix_cost が莫大なら ROI は低い（直さない方が合理的かもしれない）

**Q: 既存の info_score ユーザーへの移行は？**

A: info_score は deprecated にして delta_nats を並行表示。ダッシュボードで両方出し、一定期間後に info_score を削除。

**Q: delta-lint が finding を報告すること自体が δ を減らすのでは？パラドックスでは？**

A: パラドックスではなく、むしろ正しい動作。δ = 「暗黙の乖離」が読み手にもたらす情報損失。delta-lint が finding を報告する = 乖離を明示する = 読み手の期待を補正する = δ_読み手 ≈ 0。ただしコード自体の矛盾（δ_コード）は fix するまで残る。delta-lint は「読み手のδ」を下げる道具であり、「コードのδ」を下げるのは fix。

**Q: テストが全部通るのに δ が大きいことはありえる？**

A: ありえる。テストはカテゴリ A（矛盾系）をほぼカバーしない（§0.5.3）。「file_a と file_b のデフォルトが異なる」はどのテストも検査しない。テストが通る ≠ 矛盾がない。テストが充実したプロジェクトほど、残るバグはカテゴリ A に偏る → delta-lint の価値が上がる。

**Q: CONTEXT が大きすぎるモノレポは δ の問題ではないのか？**

A: δ ではなく μ/μ_c の問題（§1.6 参照）。μ_c = コードベースを正しく理解するのに必要な最小処理能力。モノレポで CONTEXT 巨大 → μ_c 増大 → μ/μ_c 低下 → S 低下。ただし δ=0 なら崩壊は起きない（緩やかな劣化のみ）。崩壊を起こすのは δ>0 との乗法的相互作用。

**Q: なぜ fix ベースではなくアノテーションベースの実験設計か？**

A: 3つの理由:
1. fix が正しい保証がない（fix 適用後が本当に δ=0 か検証困難）
2. アノテーション = 期待の補正 = §1.6 の δ 定義に直接対応
3. delta-lint の実際の動作（finding 報告 = 乖離の明示）と一致

**Q: δ は静的なスナップショットではないのか？**

A: 違う。δ はコード変更のたびに増減する動的量（§0.6 参照）。δ_流入（新たな矛盾の生成）と δ_流出（fix/文書化による解消）があり、δ_net = δ_流入 - δ_流出 が継続的に正のプロジェクトは構造死に向かう。scan_history.jsonl の時系列データから δ_repo(t) の推移を追跡できる。

**Q: LLM にコードを書かせてバグが出た。δ の問題か能力の問題かどう判別する？**

A: 「説明を足せば防げたか？」が判定基準（§5.3 参照）。暗黙の前提が伝わっていなかった → δ（CLAUDE.md に書けば防げた）。全情報あったが間違えた → μ/μ_c（コード複雑度の問題）。API の使い方を知らなかった → N_eff（知識の問題）。LLM の変更が新たに矛盾を生んだ → δ_流入（CI gate で捕捉すべき）。

**Q: 暗黙知を CLAUDE.md に蓄積するのは δ 低減になる？**

A: なる。§1.6 の「文書化すれば δ ≈ 0」の運用版。コードだけ見てもわからない情報（暗黙の前提、過去バグの経緯、設計判断の理由）を明文化することは、読み手の期待を事前補正する行為であり、δ を直接下げる。finding を fix するだけでなく、「なぜそうなっているか」を書き残すことが δ_流出に貢献する。

**Q: 同じパターンの finding が大量にある。全部個別に fix すべき？**

A: 必ずしも。同一パターンが特定のモジュール境界に集中している場合、個別の矛盾ではなくアーキテクチャの設計問題である可能性が高い（§7.4 参照）。18 件の④を個別に fix するより、モジュールのインターフェースを再設計した方が δ_流出が大きく、かつ今後の δ_流入も止まる。判断基準は「大改造の一時コスト < 現状維持で蓄積する δ のコスト」。Phase 4 の `diagnose` コマンドでこの判断を支援する構想。

**Q: アーキテクチャがそもそも要件に合っていない場合は？**

A: δ のスコープ外。「REST vs GraphQL の選択が間違っている」「技術スタックが要件に対して重すぎる」等はカテゴリ A（構造矛盾）ではなく、生存方程式の前提（「この系で何を作るか」）の問題。ただし、アーキテクチャ不適合の**症状**は δ として現れうる — 不適合な設計を無理に使うと矛盾が大量発生する。δ_repo が異常に高い + findings が広範囲に分散 → 「そもそもの設計戦略を見直すべき」シグナルとして読める。diagnose はこのレベルの示唆も出すことを目指す。
