広さ × 深さ

クリックでコマンドをコピー

delta-scan は 広さ（対象ファイル）× 深さ（依存の辿り方）× 質（検出手法）の3軸で構成されます。
各軸を独立に指定できます: delta-scan --scope wide --depth deep --lens security

広さ (--scope)

diff 直近3ヶ月の変更（デフォルト、--since で期間変更可）
pr ベースブランチからの全差分
smart git履歴で高リスク優先
wide 全ファイル

深さ (--depth)

指定なし 直接の依存のみ（デフォルト）
deep 依存の依存まで辿り根本から影響範囲を解析

質 (--lens)

default 構造矛盾検査（デフォルト）
stress 仮想改修で壊れ方を実験
security 認証・権限・入力検証に特化

delta-fix — 自動修正 & PR作成

検出した finding を自動修正し、ブランチ作成→デグレチェック→commit→push→PR→ステータス更新を1コマンドで実行します。

優先度上位3件を自動修正→PR
delta-fix
特定の finding を修正

GitHub Issue から修正PR作成

実行フロー: finding選択 → ブランチ作成 → fix生成 → 適用 → デグレチェック → commit → push → PR → ステータス更新

計算ロジック

技術的負債や情報量スコアの算出方法を説明。設定ファイルでカスタマイズ可能です。

技術的負債 (Technical Debt)

technical_debt = debt_coefficient × context_score

静的な重さ（負債係数）と文脈（修正の緊急度）を掛け合わせた統合指標。大きいほど優先的に対処すべき。

負債係数 (Debt Coefficient) — 0〜1.0

debt_coefficient = severity × pattern_weight × status
severityhigh=1.0 / medium=0.6 / low=0.3
pattern_weight①=1.0 ②=1.0 ③=0.9 ④=1.0 ⑤=0.8 ⑥=0.9 ⑦=0.3 ⑧=0.6 ⑨=0.5 ⑩=0.4
status$status_weights_text

確度 (Certainty) — ダブルチェック制

検出LLM（1体目）と検証LLM（2体目）の2段階判定で決定。単独の判定では「高確度」にならない。

高確度検出が high severity かつ 検証が definite + confidence ≥ 0.85
推定検証が definite/probable + confidence ≥ 0.6
未確認上記以外（検証未実施、低confidence、構造的脆弱性）

文脈スコア (Context Score)

context_score = churn_weight × fan_out_weight × user_facing × age_multiplier / fix_cost × 100
churn_weight6ヶ月の変更回数 → 0.5〜10.0（hot=月3回で最大）
fan_out_weight参照ファイル数 → 1.0〜10.0（5参照で最大）
user_facingUI/API影響=1.5× / 内部=1.0×
age_multiplier1 + log₂(1 + days/30) × churn_ratio（放置×活発=加速、churn=0なら1.0）
fix_costパターン別修正工数: ④=1.0 ⑦=0.5 ⑩=5.0 等

情報量（新規性 × 集中度）(Information Score)

info_score = discovery_value × concentration_factor × 100
discovery_value同パターン件数 n から 1/√n — リポ内で初めてのパターンほど高い（新規性）
concentration_factor同ファイルの未解決件数 m から log₂(1+m) — ホットスポットほど高い

同一リポ内の finding 分布から算出。依存・伝播は文脈スコア（fan_out）側。チーム設定に影響されない。

用途: 情報量は単体では使わず、優先度の計算に合流します。技術的負債が「この finding はどれだけ痛いか」を測るのに対し、情報量は「この finding を直すと新たに何が分かるか」を測ります。同じパターンが大量にあるファイルより、初めて見つかったパターンやホットスポットに集中した finding を優先的に修正する効果があります。

優先度 (Priority)

priority = info_score + technical_debt + severity_bonus
info_score情報量 — 新規性 × 集中度
technical_debt技術的負債 — 負債係数 × 文脈スコア
severity_bonushigh: +300 / medium: +100 / low: +0

debt_loop が自動修正の順序を決めるのに使う統合指標。「どれだけ痛いか」と「直すと何が分かるか」の両方を反映。

カスタマイズ

全ての重みは .delta-lint/config.json の scoring セクションで上書きできます。

{ "scoring": { "severity_weight": { "high": 1.0, "medium": 0.6, "low": 0.3 }, "pattern_weight": { "①": 1.0, ... }, "churn_thresholds": { "hot": 3.0, "max_weight": 10.0 }, "fan_out_thresholds": { "high": 100.0, "medium": 5.0, "max_weight": 10.0 }, "fix_cost": { "④": 1.0, "⑩": 5.0, ... } } }
